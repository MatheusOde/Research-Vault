# DSR Disciplina

## Summary

- `DSR disciplina.pdf` is teaching/protocol material that operationalizes [[wiki/concepts/Design Science Research|Design Science Research]] with maps, definitions, artifact types, evaluation modes, and research protocol prompts.
- It adapts concepts from Dresch, Lacerda, Antunes Junior, Peffers, Hevner, Gregor, and related DSR literature into practical templates.
- It treats DSR as a method for research whose goal is an artifact or prescriptive solution for a problem class.
- It expands artifact types beyond the classic construct/model/method/instantiation list by including design proposition as a reusable theoretical artifact.
- It provides a 12-step DSR protocol: problem identification, problem awareness, systematic literature review, artifact/problem-class identification, artifact proposition, artifact design, development, evaluation, learning, conclusions, generalization, communication.

## Key Claims

- A DSR project should maintain traceability among problem, artifact, theory, and results.
- Artifacts may be constructs, models, methods, instantiations, or design propositions.
- Generalization requires abstraction from the original context to a class of structurally similar problems.
- Evaluation can be observational, analytical, experimental, by test, descriptive, or through focus groups.
- Proof of concept and use case are complementary validation approaches: PoC checks technical feasibility; use case checks situated utility.

## Protocol Elements

- Identify problem: include researcher interest, problem, importance, and comprehension.
- Become aware of problem: specify context, causes when possible, expected artifact functionality, performance, requirements, and contingent heuristics.
- Review literature: identify prior artifacts and problem classes.
- Configure problem class: compare artifact alternatives and existing limitations.
- Design and develop: specify internal artifact characteristics, operation context, satisfactory solutions, construction/evaluation procedures, and expected performance.
- Evaluate: compare requirements and results, document contingent context, and formalize success/failure.
- Generalize and communicate: translate learnings into heuristics for a problem class and publish results.
